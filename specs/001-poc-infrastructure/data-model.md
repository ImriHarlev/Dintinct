## Overview

All entities fall into three categories:
1. **Fixed contracts** — defined by the spec, cannot be altered (FR-002).
2. **Internal entities** — stored in MongoDB, owned by each network.
3. **Transport objects** — Temporal signal payloads and file-based structures.

## 3. Transport / File Structures

### 3.1 File Naming Convention (Proxy Outbox)

Every file written by Network A to either outbox directory is prefixed with the `jobId` so that the Proxy Event Listener in Network B can extract the `jobId` from the filename alone — regardless of the order files arrive. This is the key to solving the chunk-before-manifest race condition.

| File type | Naming pattern | Example |
|-----------|---------------|---------|
| Data chunk | `{jobId}_chunk_{n}.bin` | `3fa85f64_chunk_1.bin` |
| Manifest | `{jobId}_manifest.json` | `3fa85f64_manifest.json` |
| Hard-fail command | `{jobId}_chunk_{n}.bin.HARDFAIL.txt` | `3fa85f64_chunk_2.bin.HARDFAIL.txt` |

The Proxy Event Listener parses the `jobId` as everything before the first `_` in the filename. With the `jobId` in hand, it calls `SignalWithStart` on `assembly-{jobId}` for every event type.

### 3.2 `Manifest` (JSON file written by Network A, read by Network B)

**File location**: Proxy Manifest Outbox directory  
**File name**: `{jobId}_manifest.json`

The manifest is a hierarchical document that gives Network B the complete reversal plan. It is the **sole source of truth** for Network B — Network B has no access to Network A's database (SC-002).

#### Schema

      "originalFormat": "pdf",

#### Root fields

| Field | Type | Notes |
|-------|------|-------|
| `JobId` | `string` | Network A's internal job ID (also the filename prefix) |
| `PackageType` | `string` | Original container type: `"zip"`, `"rar"`, `"7z"`, `"gz"`, `"directory"`, `"file"` |
| `OriginalPackageName` | `string` | Name of the original package/file (e.g., `"reports.zip"`, `"image.jpeg"`) |
| `TargetPath` | `string` | Root output path in Network B where the package is reconstructed |
| `TotalChunks` | `int` | Total chunk count across all files — used by the Assembly Workflow to know when all signals have arrived |
| `Files` | `FileDescriptor[]` | One entry per original file in the package |

#### `FileDescriptor`

| Field | Type | Notes |
|-------|------|-------|
| `OriginalRelativePath` | `string` | File's path within the original package (e.g., `"docs/report.pdf"`). Defines the output path after reverse conversion. |
| `OriginalFormat` | `string` | File extension before any conversion (e.g., `"pdf"`, `"jpeg"`) |
| `AppliedConversion` | `string` | Conversion applied by Network A: `"DirectToProxy"`, `"PNG"`, `"DOCX_AND_PNG"`, `"TXT"` |
| `ConvertedFiles` | `ConvertedFileDescriptor[]` | The file(s) that actually crossed the proxy (one for most conversions, two for `DOCX_AND_PNG`) |

#### `ConvertedFileDescriptor`

| Field | Type | Notes |
|-------|------|-------|
| `ConvertedRelativePath` | `string` | Path of the converted file relative to the package root. For `DirectToProxy`, equals `OriginalRelativePath`. |
| `Chunks` | `ChunkDescriptor[]` | Ordered list of chunks making up this converted file |

#### `ChunkDescriptor`

| Field | Type | Notes |
|-------|------|-------|
| `Name` | `string` | Chunk file name (matches the file in Network B's inbox, e.g., `3fa85f64_chunk_1.bin`) |
| `Index` | `int` | 1-based position within this converted file — defines concatenation order |
| `Checksum` | `string` | `"N/A"` in the POC mock; production value would be a checksum of the chunk bytes |

---

### 3.3 Network B Reversal Algorithm

Network B reconstructs the original package by reversing each layer of the hierarchy:

```
For each FileDescriptor in manifest.Files:
  For each ConvertedFileDescriptor:
    1. Sort chunks by Index
    2. Concatenate chunk bytes in order → produces the converted file bytes
  Apply reverse of AppliedConversion:
    - "DirectToProxy"  → write bytes directly to OriginalRelativePath
    - "PNG"            → reverse PNG conversion → write to OriginalRelativePath (e.g., .jpeg)
    - "DOCX_AND_PNG"   → merge DOCX + PNG → reverse to OriginalRelativePath (e.g., .pdf)
    - "TXT"            → reverse TXT conversion → write to OriginalRelativePath

If PackageType is "zip"/"rar"/"7z"/"gz":
  Re-pack all files at their OriginalRelativePath into the archive
  Write archive to TargetPath/OriginalPackageName
Else (PackageType is "directory" or "file"):
  Files are already written to TargetPath — no re-packing needed
```

### 3.4 Proxy Event Message (RabbitMQ → Network B)

**Format** (fixed by FR-012):

```json
{ "filePath": "/network-b/inbox/3fa85f64_chunk_1.bin" }
```

**Routing logic** (FR-013 — Proxy Event Listener inspects `filePath`):

| Condition on filename | JobId extraction | Action |
|----------------------|-----------------|--------|
| ends with `.ERROR.txt` | strip `.ERROR.txt` suffix, parse prefix | HTTP POST retry to Network A |
| ends with `.UNSUPPORTED.txt` | strip `.UNSUPPORTED.txt` suffix, parse prefix | `SignalWithStart` → `UnsupportedFileAsync` |
| ends with `_manifest.json` | parse `{jobId}` prefix | `SignalWithStart` → `ManifestArrivedAsync` |
| ends with `.HARDFAIL.txt` | parse `{jobId}` prefix | `SignalWithStart` → `HardFailAsync` |
| anything else (chunk) | parse `{jobId}` prefix | `SignalWithStart` → `ChunkArrivedAsync` |

**JobId parsing rule**: the `jobId` is the substring of the filename before the first `_` character.

### 3.5 Assembly Workflow — Aggregation Logic

#### Chunk-Before-Manifest Race Condition

Temporal buffers all signals regardless of when they arrive relative to `WaitConditionAsync`. The workflow is always safe:

1. All signal handlers (`ChunkArrivedAsync`, `ManifestArrivedAsync`, `HardFailAsync`, `UnsupportedFileAsync`) fire and update in-memory state immediately when the signal is delivered — independently of any `WaitConditionAsync` the workflow body is suspended on.
2. The workflow first awaits `WaitConditionAsync(() => _manifestReceived)`. If chunks arrived first, they are already counted in `_receivedChunkPaths`.
3. `ParseAndPersistManifestAsync` runs and returns the `AssemblyBlueprint`, which sets `_expectedChunks = blueprint.TotalChunks`.
4. The workflow then awaits the completion condition (see below). If all slots are already resolved, this returns immediately.

`SignalWithStart` guarantees the workflow exists before the first signal is delivered, so no signal is ever lost even when the listener fires before the workflow is running.

#### Completion Condition (FR-014)

A chunk slot is **resolved** when it is received normally, hard-failed, or flagged as unsupported. Hard-failed and unsupported chunks permanently close their slot and must count toward the total — otherwise the workflow waits forever for a chunk that will never arrive normally.

```
bool allArrived = await Workflow.WaitConditionAsync(
    () => _receivedChunkPaths.Count + _hardFailedChunkNames.Count + _unsupportedChunkNames.Count
          >= _expectedChunks,
    TimeSpan.FromMinutes(config.TimeoutMinutes));

// allArrived == false means the timer fired before all slots were resolved
```

## 4. Shared Domain Types — Network B (defined in `Shared.Contracts`)

`AssemblyBlueprint` is a plain C# type with no infrastructure dependencies. It lives in `Shared.Contracts` because `IAssemblyBlueprintRepository`, `IManifestStateActivities`, `IHeavyAssemblyActivities`, and `IReportingActivities` all reference it. MongoDB.Driver maps it by convention.

### 4.1 `AssemblyBlueprint` (Network B — derived from manifest — `Shared.Contracts`)

The blueprint is a persisted copy of the manifest enriched with runtime tracking state. It is the source of truth for the assembly progress.

| Field | Type | Notes |
|-------|------|-------|
| `Id` | `string` | MongoDB `_id` — same as `Manifest.JobId` |
| `PackageType` | `string` | From manifest — drives re-packing logic |
| `OriginalPackageName` | `string` | From manifest |
| `TargetPath` | `string` | From manifest — root output path |
| `TotalChunks` | `int` | From manifest — total expected chunk count |
| `Files` | `FileDescriptor[]` | Full hierarchy from manifest — drives the reversal algorithm |
| `ReceivedChunkNames` | `HashSet<string>` | Chunk names that have arrived (keyed by `ChunkDescriptor.Name`) |
| `UnsupportedChunkNames` | `HashSet<string>` | Chunk names flagged as NOT_SUPPORTED |
| `HardFailedChunkNames` | `HashSet<string>` | Chunk names permanently failed via `.HARDFAIL.txt` |
| `Status` | `string` | `"Aggregating"`, `"Assembling"`, `"Completed"`, `"Failed"`, `"Timeout"` |
| `CreatedAt` | `DateTime` | UTC |
| `UpdatedAt` | `DateTime` | UTC |

**Collection**: `assembly_blueprints`  
**Indexes**: `Id` (primary, same as jobId)  
**Repository interface**: `IAssemblyBlueprintRepository`  
**Note**: `Files` uses the standalone `FileDescriptor` / `ConvertedFileDescriptor` / `ChunkDescriptor` types defined in `Shared.Contracts` (see `contracts/interfaces.md` Shared Value Types section).

## State Transition Diagrams

### Job (Network A)

```
Received → Processing → AwaitingCallback → Completed
                    ↘ Retrying ↗
                    ↘ Failed (hard)
                    ↘ Timeout
```

### AssemblyBlueprint (Network B)

```
Aggregating → Assembling → Completed
                        ↘ CompletedPartially
                        ↘ Failed
           ↘ Timeout (workflow timeout exceeded)
```
